# ReportView 重构与 DCF 工具页设计

日期：2026-04-12

## 1. 目标

本次改造包含两个前端目标：

1. 将 `frontend/src/components/ReportView.tsx` 从“原始 agent 输出展示页”重构为“答辩可用的主题驾驶舱”。
2. 新增一个独立的 DCF 估值工具页，用于展示 DCF 假设如何影响估值结果，并允许手动调参与演示。

设计原则：

- 主页面先给结论，再解释理由。
- 不把“缺数据”和“看空/看多”混为一谈。
- DCF 不能作为黑箱结果展示，必须解释参数取值及其含义。
- 多智能体协作要保留，但只能作为次级展示层，不应压过最终结论。

## 2. 非目标

- 本轮不改 `AnalysisStatus.tsx` 的阶段可视化主结构。
- 本轮不引入真实交易/个性化仓位建议输入。
- 本轮不重新设计全站导航与整体视觉系统，只在现有样式体系上扩展。
- 本轮不修改后端 DCF 算法逻辑；前端先以现有结果与已有数据结构为基础做展示与参数覆盖计算。

## 3. ReportView 信息架构

`ReportView` 重构为六层结构：

### 3.1 最终建议区

页面第一视觉中心，直接回答“这只股票当前应如何看待”。

展示内容：

- 最终建议：`买入 / 观望 / 回避`
- 一句核心理由
- 股票代码
- 结论强弱/置信度
- 数据完整度标签

不展示：

- 建议股数
- 建议仓位

原因：

- 项目没有个人约束输入，仓位与股数会造成“看似精确、实际不成立”的误导。

### 3.2 为什么这么判断

用三块摘要面板解释决策形成过程：

- 支持理由
- 反对理由
- 最终平衡判断

数据来源将聚合：

- `researcher_bull`
- `researcher_bear`
- `debate_room`
- `portfolio_manager`

这一层不按 agent 名称叙事，而是按“观点冲突如何被平衡”来叙事。

### 3.3 风险与可靠度

拆成两个独立模块：

- 风险提示：波动、回撤、消息面不确定性、情绪脆弱性
- 数据可靠度：关键数据是否完整、哪些模块因缺数据而降级

关键要求：

- 如果财务指标、财务报表、市场数据缺失，必须明确提示“本轮结论应保守解释”。
- `N/A`、空值、`0` 不直接暴露给用户，而要翻译成人话。

### 3.4 四大分析模块

主视图固定为四个主题模块：

- 估值面
- 基本面
- 消息/情绪面
- 风险面

每个模块统一输出：

- 模块结论
- 一句解释
- 2 到 4 个关键证据

示例：

- 估值面不是直接展示 `intrinsic_value=0`，而是写成“本轮估值结论为数据不足，原因是自由现金流或市值输入无效，DCF 无法形成可靠输出”。

### 3.5 多智能体观点区

保留多 agent 协作展示，但降级到主视图之后。

每张卡片仅保留：

- agent 身份
- 倾向
- 摘要
- 置信度

详细字段折叠显示，避免主页面被技术字段淹没。

### 3.6 原始数据区

提供折叠面板：

- 原始 JSON
- 原始 agent 字段
- 详细报告文本

默认折叠，仅供调试、补充说明和答辩备用。

## 4. 缺数据表达规范

缺数据场景采用“双层表达”：

- 标题层：明确标记 `数据不足`
- 正文层：解释“因此本项暂时无法形成可靠判断”

统一替换规则：

- `N/A` -> `暂无有效数据`
- `0` 且语义为“模型无结果” -> `本轮不输出结论`
- 空数组/空字段 -> `暂无支持证据`

示例：

- `PB 历史缺失` -> `相对估值所需的 PB 历史数据缺失，因此本轮无法判断估值处于高位还是低位`
- `DCF intrinsic_value=0` -> `DCF 所需关键输入无效，因此本轮 DCF 不可用，不能据此判断低估或高估`

## 5. DCF 展示设计

DCF 在页面中分两层展示：

### 5.1 ReportView 中的 DCF 摘要

估值模块中固定展示：

- DCF 是否可用
- DCF 结论
- 三个关键参数

固定展示的三个参数：

- 折现率
- 第一阶段/未来现金流增长率
- 永续增长率

参数展示形式：

- 参数值
- 一句解释“这意味着什么”

示例：

- `折现率 10%：表示模型对风险和资金成本采用偏保守假设`

### 5.2 DCF 假设详情

在报告页中保留跳转入口，进入独立 DCF 工具页查看与调参。

如果 DCF 不可用，必须显示：

- 不可用状态
- 不可用原因
- 缺失的关键输入

不允许把 `0` 当作估值结果展示。

## 6. 独立 DCF 工具页设计

### 6.1 页面职责

该页面是一个独立工具页，不依赖某次分析结果才能工作，但支持从报告页跳转并自动带入当前分析结果中的股票代码和基础数据。

定位：

- 上半部：交互演示
- 下半部：计算依据与参数解释

### 6.2 数据填充策略

默认行为：

- 自动带入当前分析结果中的股票代码和已有基础数据

同时允许：

- 所有关键输入手动覆盖

这保证它既能承接当前分析，也能单独作为答辩演示工具。

### 6.3 输入设计

默认主操作区开放核心参数：

- 股票代码
- 当前价格
- 自由现金流
- 第一阶段增长率
- 第二阶段增长率
- 永续增长率
- 折现率
- 预测期年数

高级设置折叠开放扩展参数：

- 总股本/流通股本
- 当前市值
- 税率
- 净债务
- Beta
- 权益风险溢价
- 无风险利率
- 资本结构
- 债务成本
- 终值口径

原则：

- 页面支持“全量可改”
- 但默认不把所有参数同时铺开，避免界面失控

### 6.4 输出设计

页面右侧或顶部结果区实时展示：

- 内在价值
- 每股估值
- 安全边际
- 当前价格对比
- 估值结论

同时提供“参数变化影响”提示：

- 当前结果偏保守/偏乐观
- 哪个参数最敏感

### 6.5 依据展示区

下半部展示：

- DCF 公式口径说明
- 参数解释
- 数据来源说明
- 关键中间值
- 结果为何对某些参数敏感

目标不是做学术公式推导页，而是让老师理解：

- 这个估值不是唯一答案
- 它取决于假设
- 项目已经把这种主观性显式暴露出来

## 7. 组件与实现边界

建议拆分前端组件，避免 `ReportView.tsx` 再次膨胀：

- `ReportView.tsx`
- `report/DecisionHero.tsx`
- `report/ReasonSummaryPanel.tsx`
- `report/RiskReliabilityPanel.tsx`
- `report/AnalysisModuleGrid.tsx`
- `report/AgentPerspectiveSection.tsx`
- `report/RawDataDrawer.tsx`
- `valuation/DcfWorkbenchPage.tsx`
- `valuation/DcfParameterForm.tsx`
- `valuation/DcfResultPanel.tsx`
- `valuation/DcfAssumptionNotes.tsx`

如果当前项目路由结构不便大拆，可先在 `components/` 下组织，但仍需保持模块边界清晰。

## 8. 数据整理层

前端需要增加一层展示用转换逻辑，而不是让组件直接消费后端原始字段。

建议新增：

- `frontend/src/utils/reportView.ts`
- `frontend/src/utils/dcfView.ts`

职责：

- 翻译原始信号为中文结论
- 统一处理缺数据文案
- 从多个 agent 输出中聚合主题模块
- 提取 DCF 参数与结果
- 生成“支持理由 / 反对理由 / 最终平衡”摘要

## 9. 视觉方向

视觉目标不是“更多卡片”，而是“更强层级”：

- 顶部大结论区做明显视觉中心
- 主题模块用稳定网格呈现
- 风险与可靠度单独强调
- agent 区和原始数据区弱化

视觉要求：

- 继续沿用现有页面视觉系统，不另起一套主题
- 强化标题层级与卡片主次关系
- 适配桌面答辩主场景，同时保证移动端不崩

## 10. 错误与边界情况

需要显式处理：

- 无分析结果
- 只有部分 agent 输出
- DCF 参数缺失
- DCF 结果无效
- 当前分析与工具页输入不一致

对应策略：

- 页面不崩溃
- 用明确提示解释原因
- 不输出伪精确结论

## 11. 验证方案

前端改造完成后至少验证：

1. `ReportView` 在完整数据下能正确显示六层结构
2. 缺财务/估值数据时，页面输出“数据不足”而不是裸 `N/A`
3. DCF 不可用时，主视图与工具页都能解释原因
4. DCF 工具页改参数后，结果区实时更新
5. 从报告页进入 DCF 工具页时可自动带入当前股票与基础数据
6. `npm run build` 通过

## 12. 范围判断

本次设计范围适合作为一个实现计划执行，不需要继续拆成独立子项目。

优先级顺序：

1. 重构 `ReportView`
2. 新增展示用数据整理层
3. 新增独立 DCF 工具页
4. 接通报告页到 DCF 工具页的跳转
